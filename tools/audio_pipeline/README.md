# `tools/audio_pipeline/` — manifest-driven audio pipeline

One manifest is the single source of truth for Qalam's bundled pronunciation
clips. It replaces the old hand-maintained lockstep between
`assets/audio/README.md` and the `_audioIdToAsset` map in
`lib/services/asset_audio_player.dart` — which would not have survived 28 letters
× N words.

## The one source of truth

[`audio_manifest.json`](audio_manifest.json) — one entry per clip:

| field | meaning |
|-------|---------|
| `audioId` | the Schema-v2 id the app resolves (`snd.<letterId>` / `word.<id>` / `sentence.<id>`) |
| `assetPath` | repo-relative bundled path, always under `assets/audio/` |
| `description` | what the clip says |
| `usedBy` | where it's referenced today (informational) |
| `status` | `real` \| `placeholder` \| `draft-tts` |

**`status` is honest.** `real` = the owner's-mother's recording. `draft-tts` =
interim AI voice (all 39 clips today; generated by `tools/tts/generate_audio.py`).
`placeholder` = silent stand-in. A clip is only `real` once a human confirms it.

## Commands

Run from the `tools/` directory:

```bash
python -m audio_pipeline generate   # rewrite the README table + Dart map from the manifest
python -m audio_pipeline check      # exit non-zero on ANY drift (use in CI)
python -m audio_pipeline normalize  # ingest + normalize raw recordings from staging/
```

- **`generate`** rewrites only the content between the
  `BEGIN GENERATED (audio_manifest)` / `END GENERATED` markers in
  `assets/audio/README.md` and `asset_audio_player.dart`. Everything else in
  those files is hand-authored and never touched.
- **`check`** fails if a manifest entry has no file, a clip file has no entry, or
  either generated block is stale. Zero mutations — safe in CI.
- **`normalize`** needs [`ffmpeg`](https://ffmpeg.org/download.html) on PATH.

## Recording → bundled clip workflow

1. Record the clip (short, clean). Any sane format (`wav`/`m4a`/`mp3`/`flac`/…).
2. Name it `<audioId>.<ext>` — e.g. `snd.baa.wav` — and drop it in
   [`staging/`](staging/) (gitignored; raw recordings are not committed).
3. `python -m audio_pipeline normalize` — trims leading/trailing silence,
   loudness-normalizes to EBU R128 (~ −16 LUFS, mono), resamples to 44.1 kHz,
   encodes mp3, writes to the manifest's `assetPath`.
4. If the clip is a **real** human recording, set its `status` to `real` in
   `audio_manifest.json`.
5. `python -m audio_pipeline generate` then `python -m audio_pipeline check`.

The pipeline is **voice-agnostic** — whose voice records the real clips is the
owner + mother's decision. It never relabels a `draft-tts` clip `real` for you,
and never guesses: a staging file whose stem isn't a known audioId is reported
and skipped.

## Adding a brand-new clip

Add an entry to `audio_manifest.json` (new `audioId` + `assetPath` under
`assets/audio/`), drop the recording in `staging/` (or place the file directly),
then `generate` + `check`. `assets/audio/` is already declared in
`pubspec.yaml`, so a new file bundles on the next build.

## Relationship to `tools/tts/`

`tools/tts/` **synthesizes** interim AI-voice clips (ElevenLabs). This pipeline
**normalizes real recordings** and is the **source of truth for the
audioId → asset mapping** (README + Dart map + status). They are complementary;
this pipeline does not call the TTS tool.
