# ADR-014: v2 AI-Tutor Agent Architecture & Topology

**Status:** Topology SUPERSEDED by [ADR-015](ADR-015-v2-tutor-server-langgraph-agent.md) (owner, 2026-06-22) ŌĆö the agent now runs **server-side on a LangGraph framework**, not client-only, after the agent's scope grew from "one coaching line" to a capable analyze/plan agent. **Everything else here still holds:** the grounding invariant, the 4 ACTION tools, FACTS-in/ACTIONS-out, the `TutorBrain` seam, and the `AuthoredFallback` offline floor. Read ADR-015 for the current topology + framework.
**Date:** 2026-06-22
**Deciders:** Owner (final call on pedagogy/product), with flutter-expert + ai-engineer input
**Supersedes/affects:** Phase 11 spike findings (genui drop holds); reshapes Phase 14 plan; constrains Phases 12/13/16.

---

## Context

Qalam v2 adds an AI tutor for ONE letter family (baa) to an Android Flutter app that already does deterministic on-device geometric stroke scoring. The tutor must be warm, specific, child-pitched, and ŌĆö critically ŌĆö must NOT touch the existing v1 canvas, scorer, or curriculum. Hard constraints (owner, 2026-06-21):

- Client-side model only: **Gemini** (cloud, via Firebase AI Logic) and **Gemma** (on-device). Claude is ruled out (no client proxy via Firebase AI Logic). API key NEVER in the client (Firebase AI Logic + App Check enforced in prod).
- **Grounding invariant:** the deterministic scorer owns pass/fail and the mastery star; the agent only chooses WORDS and the next exercise. Verdict + learner state are injected as FACTS, never exposed as tools. The agent acts ONLY through 4 ACTION tools: `present_activity`, `say`, `give_hint`, `advance`.
- **Non-PII guard:** only derived facts (`mistakeId` enum, struggle tags, `letterId`) may cross the network ŌĆö never raw strokes, never nickname/PII.
- **Offline floor:** an `AuthoredFallback` (owner-signed coaching lines) must work with zero model, airplane mode.
- Local Drift for progress (no Firestore child data). Android-only. Anti-gamification (one quiet star). Reuse v1 layers UNTOUCHED.
- Technion course project: no ops budget, a live demo deadline, owner is new to Dart and wants explainable, low-magic code.

The topology was declared GENUINELY OPEN, weighing three equal contenders: **Path A** (client-only: Flutter ŌåÆ Gemini/Gemma directly), **Path B** (a thin agent backend running a framework and speaking AG-UI/MCP/A2UI to Flutter), and **Path C** (staged: ship A, design a seam so B can slot in later). Depth was exhaustive: every "supports Flutter/Dart" claim was verified against pub.dev + a real repo, with recency noted (these packages ship ~monthly; pre-1.0 churn already bit us with genui 0.9.2 in the Phase 11 spike).

**Verified facts (pub.dev, 2026-06-22):**
- `firebase_ai` **3.13.0**, published 4 days ago ŌĆö first-party, stable, supports function declarations + `ToolConfig`/`FunctionCallingConfig`.
- `flutter_gemma` **1.0.2**, 392 likes, ~18.4k weekly downloads ŌĆö community but real; native `<tool_call>` tokens, JSON-Schema tools, `toolChoice`, fully offline.
- `ag_ui` (the ONLY AG-UI Dart client) **0.1.0**, published 9 months ago, **6 likes**, explicitly backend-bound (examples init against `http://localhost:8000`); maintainers declined a client-only mode on security grounds. The AG-UIŌåöA2UI bridge is React/Next.js only ŌĆö zero Flutter touchpoint.
- `genui`/A2UI still 0.9.x alpha; the Phase 11 canvas-State-teardown risk is unfixed and its core design (stream widget state back to the LLM, a "high-bandwidth loop") is the architectural opposite of our two-clocks reflex and grounding invariant.
- **The brief's unifying-seam hope is FALSE for Flutter:** Firebase AI Logic *hybrid* on-device inference has no Dart SDK; its on-device leg is platform-fixed (Gemini Nano, not Gemma) and does NOT support function calling. Cloud and on-device are therefore TWO seams, not one.

## Decision

Adopt **Path A, client-only**, re-scoped exactly per the adversarial review to **"pure-A plus one paragraph of documented foresight"** (this is the honest, surviving form of "Path C"; see *Alternatives*). No server, no AG-UI/A2UI/MCP, no genui.

### 1. Topology
Flutter talks to the model directly. $0 ops, one network hop, lowest latency, smallest child-data trust boundary, offline floor real today. This is exactly where shipped Flutter AI apps sit (Reflection.app on flutter.dev's showcase; the flutter.dev/ai recommended stack).

### 2. The agent loop ŌĆö GeminiBrain (Path-A baseline)
- `FirebaseAI.googleAI().generativeModel(model: <gemini Flash ŌĆö confirm exact string at build>, tools: [4 FunctionDeclarations], toolConfig: ToolConfig(functionCallingConfig: FunctionCallingConfig.any({'present_activity','say','give_hint','advance'})))`. `.any({4})` pins the action space to exactly our 4 tools ŌĆö a grounding lever, not a guideline.
- Loop = `startChat()` ŌåÆ `sendMessage(Content.text(systemInstruction + FACTS))` ŌåÆ `while (response.functionCalls.isNotEmpty) { dispatch; sendMessage(Content.functionResponses(...)); }`. This is the official flutter.dev crossword pattern ŌĆö ~30 lines, explainable to a Dart-new owner.
- **FACTS are injected as system-instruction / Content TEXT, never as a `responseSchema`** (verified: structured-output + function-calling throws an `application/json` mime exception). This reinforces the grounding shape: FACTS-in-as-text, ACTIONS-out-as-tools.
- **ACTION turns run NON-streamed** (`chat.sendMessage`); streaming + tool-calls is undocumented on Dart. If we want streamed warmth, isolate it to the final, non-tool `say()` turn (or Live API later) ŌĆö carried as a revisit trigger, not built now.
- Bounded manual retry-with-backoff around `sendMessage` (no built-in retry).
- Key never in client: firebase_ai proxies via Firebase AI Logic. **App Check MUST be enforced in prod (Phase 14)** with limited-use/replay-protection tokens; Phase 11 left it off for the throwaway spike.

### 3. Agent Ōåö UI ŌĆö native function-call dispatcher (NO protocol)
A ~100-line `switch (functionCall.name)` maps each ACTION tool to an imperative call on a native controller (`canvasController.presentActivity(...)`, `tutorPanel.say(...)`, etc.). **The `StrokeCanvas` stays a plain `StatefulWidget` OUTSIDE any reactive/agent-driven rebuild scope** ŌĆö dispatch imperatively, never rebuild the canvas from agent state. This single rule preserves the two-clocks reflex and is the Phase 11 kill-shot lesson encoded. Guard it in code review.

### 4. On-device / offline ŌĆö TWO seams, behind ONE local abstraction
- **The seam (`TutorBrain`):** `abstract class TutorBrain { Future<TutorDecision> next(TutorFacts facts); }`. Its justification is the **offline floor + the on-device bake-off router** ŌĆö three implementations that all ship in v2 ŌĆö NOT a future backend.
  - `TutorFacts` (FACTS-in): a whitelisted non-PII DTO built by ONE chokepoint function ŌĆö `{letterId, mistakeId (enum), struggleTags, pass/fail, sessionHistory-derived}`. Raw `List<Offset>` strokes and nickname/PII never reach this builder. A unit test asserts the serialized payload contains only enum/id fields (GROUND-02).
  - `TutorDecision` (ACTIONS-out): one of the 4 ACTION tool calls. Verdict + star are NOT tools; the scorer owns them and injects them as FACTS (GROUND-01).
  - **GeminiBrain** (cloud, the demo baseline), **GemmaBrain** (`flutter_gemma` 1.0.2, on-device, fully offline, native tool tokens ŌĆö **experimental, gated on the Phase 13 bake-off result**), **AuthoredFallback** (pure Dart, airplane-mode floor, the guaranteed degrade).
- **Router** (in the TutorBrain factory): online + capable ŌåÆ GeminiBrain; offline / experimental-on-device ŌåÆ GemmaBrain; no model / failure ŌåÆ AuthoredFallback. Because the scorer owns the verdict, on-device weakness degrades only to "less apt coaching," never to "wrong pass/fail."
- Durable v1 layers (canvas / scorer / curriculum) carry ZERO `firebase_ai` / `genui` / `flutter_gemma` imports.

### 5. Path B is documented, NOT built (the re-scope the refutation forces)
Per the adversarial review, the `next(facts)` interface is the correct seam for the three IN-PROCESS brains but is explicitly **NOT** the transport contract a remote backend needs (event streaming, SSE ordering, cancellation on canvas reset, per-request App Check, connection lifecycle). We therefore do NOT claim a 4th impl drops in behind it. **The only Path-B artifact created now is prose in this ADR.** No abstraction, factory hook, transport stub, or `RemoteBrain` placeholder is written. If/when the migration trigger fires, the future seam is **Genkit-flow-over-HTTP** (lower-magic than AG-UI), and the AG-UI-to-Flutter framing should be retired until its Dart SDK is real.

### How the grounding invariant + non-PII guard are enforced (summary)
1. Action space pinned by `FunctionCallingConfig.any({4 names})` ŌĆö the model literally cannot emit a 5th action.
2. Verdict/star never appear as tools; they are FACTS-as-text. The dispatcher has no "set verdict" branch.
3. One DTO chokepoint builds `TutorFacts`; a unit test asserts no raw strokes / no PII in the serialized payload (GROUND-02 test).
4. The dispatcher is a closed `switch` over 4 names; any unrecognized call is a logged no-op, not a crash.

## Consequences

**Good**
- Ships on the Technion timeline: ~30-line loop + ~100-line dispatcher, no server, no protocol, no ops budget, explainable to a Dart-new owner.
- Lowest latency (one hop, ~0.6ŌĆō0.8s Gemini TTFT), smallest child-data trust boundary, real offline floor today.
- Grounding invariant enforced by construction (pinned tool set + FACTS-as-text + one chokepoint), unit-testable.
- v1 canvas/scorer/curriculum untouched; the imperative-dispatch rule preserves the two-clocks reflex.
- The local `TutorBrain` seam earns its keep on shipping grounds (offline floor + on-device router), independent of any future backend.

**Bad / accepted costs**
- We hand-roll orchestration; no framework multi-step sophistication. Acceptable: the agent only "picks words + chooses next exercise" for one letter family.
- `firebase_ai` ships ~monthly and `flutter_gemma` just crossed 1.0 ŌĆö both must be re-pinned at Phase 14 plan time and the exact Gemini model string confirmed at build.
- GemmaBrain's on-device Arabic + 4-tool quality is UNPROVEN on external evidence (the TimeLens "Gemma 4 E2B clean Arabic" result ran the LLM server-side) and FunctionGemma-270M out-of-box tool accuracy is 10ŌĆō58%; GemmaBrain must stay experimental until the Phase 13 bake-off measures it on a real device.
- Device cost for GemmaBrain is real (~2.5GB download, 8GB-RAM-class, arm64, first-run download UX) and may keep it experimental-only regardless of quality.
- We consciously forgo the 2026 protocol-stack momentum (AG-UI/MCP/A2UI) ŌĆö but that momentum is a server+React+web story, not a Flutter-mobile one yet; we keep the door noted, not open in code.
- Streaming the `say()` turn is deferred; if warmth needs it, it is isolated work later.

## Alternatives considered

**Path A ŌĆö pure client-only (CHOSEN, in its re-scoped form).** firebase_ai + flutter_gemma + a Dart loop + native dispatcher. Satisfies every hard constraint by construction. The only thing it "lacks" is framework orchestration we do not need.

**Path B ŌĆö thin agent backend speaking AG-UI/MCP/A2UI to Flutter (REJECTED for v2).** Why not: the entire 2026 protocol stack presupposes a running agent server. The ONLY AG-UI Dart client (`ag_ui`) is frozen at 0.1.0, 9 months stale, 6 likes, backend-bound, and cannot run client-only. The AG-UIŌåöA2UI bridge is React-only. MCP is overkill for a fixed curriculum with 4 local Dart tools and no second tool-consumer. B reintroduces the deliberately-deferred server (Cloud Run min-instances=1, deploy/secret/monitoring), a per-coaching-turn network hop + cold-start risk on top of Gemini's TTFT, and a child-data-on-the-wire surface ŌĆö all on a Technion demo deadline, for a loop whose every framework-bought capability (multi-step orchestration, tool ecosystems, shared server state) the product explicitly does not need.

**Path C ŌĆö "ship A, build a seam so B slots in later" (REJECTED as originally framed; SURVIVES only re-scoped ŌĆö which is what we chose).** The adversarial review holed the *original rationale*: the `Future<TutorDecision> next(TutorFacts)` interface that cleanly unifies the three LOCAL brains is provably NOT the streaming/transport contract a remote backend needs (it has no notion of token streaming, SSE ordering, connection lifecycle, partial/aborted turns, cancellation, or per-request App Check). So "a 4th impl slots in behind the same interface" is a false promise; day-B would redesign the interface anyway. Paying any abstraction tax NOW for B would make C strictly worse than pure-A on owner comprehension and demo-hardening time. C survives ONLY when re-scoped so its "design for B" cost is literally zero beyond pure-A: (1) the `TutorBrain` seam is justified by the local three brains, never by B; (2) the only B-artifact now is prose (this ADR section); (3) the seam must earn its keep on local grounds or be dropped (if the on-device `<tool_call>` path or a streamed final-say turn forces the interface to diverge before the demo, simplify to direct firebase_ai and gate Gemma to Phase 13). Re-scoped, C *is* "pure-A with a paragraph of foresight" ŌĆö which dominates both pure-A (door honestly noted) and B (no server, no churn, no magic). **That re-scoped form is the Decision above.**

**Protocols, why-not (each):** *genui/A2UI* ŌĆö DROP holds (0.9.x alpha, unfixed canvas-State gap, high-bandwidth-loop design opposes our grounding). *ag_ui* ŌĆö RULE OUT (Path-B-only, 0.1.0/9mo-stale/6-likes, backend-bound). *MCP* ŌĆö RULE OUT (overkill for 4 fixed local tools; no Firebase-AI transport; key-in-client risk). *Flutter AI Toolkit* ŌĆö SKIP (chat-bubble UI, wrong shape). *Firebase AI Logic hybrid* ŌĆö RULE OUT for v2 (no Dart SDK, on-device leg is Gemini-Nano-only with no function calling). *Genkit Dart* ŌĆö DEFER (preview/pre-1.0; the days-old on-device plugin has 0 likes) ŌĆö named only as the *future* Path-B-Lite seam. *LangGraph/ADK/CrewAI* ŌĆö OUT (no Flutter client, server-side).

## Revisit triggers

1. **Re-open Path B** only if Qalam grows (a) a genuinely agent-composed FORM-like surface distinct from the canvas (e.g. a parent dashboard), OR (b) external tools / a second tool-consumer, OR (c) server-side multi-step memory. Then add a `RemoteGenkitFlowBackend` as a NEW (streaming/transport) interface ŌĆö Genkit-flow-over-HTTP, not AG-UI ŌĆö and accept that the local `TutorBrain` interface does not transfer.
2. **Re-pin dependencies** at Phase 14 plan time (`flutter pub outdated`); confirm the exact Gemini Flash model string (naming moved past `gemini-2.5-flash`).
3. **Gate GemmaBrain** on the Phase 13 bake-off: measure 4-tool selection accuracy AND Arabic register ON A REAL DEVICE. Do not ship on external evidence.
4. **Measure on-device latency** on the actual target Pixel Tablet in Phase 12 (the 10ŌĆō25 tok/s figure is from secondary sources / other devices only).
5. **Streaming say():** if warmth needs streamed text, isolate it to the final non-tool turn or Live API ŌĆö revisit then, not now.
6. **Re-open genui/A2UI** ONLY if A2UI reaches stable v1.0 AND genui ships a documented State-preservation / stable-identity contract for hosted stateful widgets AND the product grows an agent-composed surface distinct from the canvas.
7. **Canvas-rebuild regression:** if anyone rebuilds `StrokeCanvas` from agent/reactive state, the Phase 11 kill-shot re-applies ŌĆö treat as a release blocker.
