# Phase 06.1 Plan 04: Firestore Curriculum Repository Summary

**`CurriculumRepository` now reads letters, lessons, and the tolerance ramp live from Firestore via a one-shot `.get()`, falls back to the bundled JSON on a cold/empty/throwing Firestore so the app still works offline, validates whichever source won before caching, and serves everything from the kept-alive in-memory cache — all behind the unchanged async-getter API.**

## Performance

- **Duration:** ~20 min
- **Completed:** 2026-06-14
- **Tasks:** 2 (Task 1 auto; Task 2 TDD: RED -> GREEN)
- **Files:** 4 modified, 1 created

## Accomplishments

- **Live Firestore read with offline bundle fallback (D-01/D-02/D-04).** `_ensureLoaded()` now branches by source: the `.fromStrings` JSON-override path is untouched; the live path does a one-shot `await _firestore.collection('letters').get()` (and `lessons`, and `meta/toleranceRamp`), maps each doc through the Plan-02 codec, and falls back to `rootBundle.loadString('assets/curriculum/*.json')` on empty/exception (cold-first-run). The 28-letter bundle floor is proven for both empty and throwing Firestore.
- **Validator runs over the chosen source (D-05, T-06.1-10).** The existing `validateReferenceStrokes` D-04 guard runs over every letter regardless of source. A closed-loop stroke seeded into Firestore makes `getLetters()` throw `StateError("...closed outline loop...")`; the cache stays null on throw, so a second call re-runs and re-throws — the invalid stroke never reaches the scorer.
- **`.withFirestore` injection seam + preserved `.fromStrings` (D-04).** A new `CurriculumRepository.withFirestore(FirebaseFirestore)` constructor injects a `FakeFirebaseFirestore`. `FirebaseFirestore.instance` is resolved lazily, so `.fromStrings` constructs no Firebase instance — the bundle/JSON tests stay both network-free and Firebase-free (no Firebase app needed in the test env).
- **Ramp from the meta doc with a defensive chain (D-07, Pitfall 5).** `getDefaultToleranceRamp()` reads `meta/toleranceRamp` (field `ramp`) first; absent/empty -> the bundle's `defaultToleranceRamp` -> the decided `['loose','normal','strict']`. Never throws.
- **Offline-first preserved (D-03/PLAT-01).** The read happens once at the first getter call into the kept-alive provider cache; the practice/scoring path is cache-served thereafter and never awaits the network. All six public getter signatures are byte-for-byte unchanged — call sites elsewhere are untouched (full suite green apart from documented pre-existing failures).

## Decisions Made

- **fake_cloud_firestore over the mocktail fallback.** `flutter pub add --dev fake_cloud_firestore` resolved `4.1.1` cleanly against the installed `cloud_firestore 6.5.0` — research caveat A1 (possible version mismatch) did not materialize, so the richer in-memory fake was used rather than hand-rolled mocktail stubs.
- **Lazy `FirebaseFirestore.instance`.** The default constructor stores an optional override and resolves `FirebaseFirestore.instance` only on first read (via a getter). This keeps `.fromStrings` from triggering `[core/no-app]` in the test environment, which has no initialized Firebase app — discovered during Task 1 (see Deviation below).
- **Ramp fallback chain.** Firestore meta doc -> bundle ramp -> decided default. Reads the bundle `lessons.json` for the ramp independently of the lessons collection result, so the ramp is correct even when lessons come from Firestore but no meta doc was seeded.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Eager `FirebaseFirestore.instance` in `.fromStrings` crashed the bundle tests**
- **Found during:** Task 1 (first run of the existing `curriculum_repository_test.dart`).
- **Issue:** The initial Task-1 edit initialized `_firestore = FirebaseFirestore.instance` in every constructor (including `.fromStrings`). In the test environment no Firebase app is initialized, so constructing the instance threw `[core/no-app] No Firebase App '[DEFAULT]' has been created`, failing 28 previously-passing bundle tests.
- **Fix:** Made `_firestore` a lazy getter (`_firestoreOverride ??= FirebaseFirestore.instance`) resolved only when the Firestore read path actually runs. `.fromStrings` leaves the override null and never touches it.
- **Files modified:** `lib/data/curriculum_repository.dart`
- **Commit:** `b6fc719`

**2. [Rule 2 - Security] Admin-SDK service-account key was not git-ignored**
- **Found during:** Task 1 (`git status` before staging showed an untracked `tools/firebase/qalam-app-bd7d0-firebase-adminsdk-fbsvc-e4c937313d.json`).
- **Issue:** The existing `.gitignore` patterns (`*serviceAccount*.json`, `*.key.json`) did NOT match the admin-SDK key's default filename (`*adminsdk*.json`), risking a private-key commit (threat T-06.1-02, elevation-of-privilege).
- **Fix:** Added `tools/firebase/*adminsdk*.json` and `tools/curriculum/*adminsdk*.json` to `.gitignore`; confirmed `git check-ignore` now matches the key. The key is NOT committed.
- **Files modified:** `.gitignore`
- **Commit:** `b6fc719`
