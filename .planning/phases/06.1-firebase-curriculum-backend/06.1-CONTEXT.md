# Phase 06.1: Firebase Curriculum Backend - Context

**Gathered:** 2026-06-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Make **Firestore** (in the existing `qalam-app-bd7d0` project) the cloud source-of-truth
for curriculum content — letters and lessons — and wire the Flutter app to read it **live
at runtime** with offline persistence, backed by a **bundled seed snapshot** for cold
first-run so the offline-first promise survives. Migrate the signed-off **alif** + the
28-lesson skeleton into Firestore. Stand up **Firebase Auth (anonymous)** so security rules
can gate reads.

**In scope:** FlutterFire wiring (firebase_core, cloud_firestore, firebase_auth,
`firebase_options.dart`, `google-services.json`); Firestore `letters` + `lessons`
collections (doc-per-letter, nested); a **full, v2-ready Firebase Auth foundation** —
real providers enabled (Anonymous + Email/Password + Google), v1 runtime uses anonymous,
anonymous→permanent **account linking** architected, role-ready security rules; Firestore
offline persistence; a bundled seed snapshot for first-run; a Python `firebase-admin` seed
script that loads alif + lessons; round-trip parity with the existing Dart models.

**Out of scope:** authoring the other 27 letters and grammar/sentence questions (Phases 7–8,
owner's-mother sign-off); Cloud Functions / the AI tutor (v2); a custom in-app authoring UI;
**any child login UI or child account/PII** (Decided child-safety — children stay anonymous);
App Check (Phase 10 hardening); child profile/progress data in Firestore (stays local Drift
per Phase 5/6).

**⚠ Owner override (2026-06-14):** This phase reverses the Decided "v1 local-only, no
Firebase" item (PROJECT.md / STATE.md line 86). Confirmed by the owner. Offline-first
(PLAT-01) is preserved by the bundled-seed + Firestore-persistence requirement below, not
by avoiding cloud.

<decisions>
## Implementation Decisions

### Runtime data source (the offline contract)
- **D-01:** The app reads curriculum **live from Firestore** at runtime (owner's choice over
  bundle-only), NOT from the bundled JSON as the primary path.
- **D-02:** **[HARD CONSTRAINT — preserves PLAT-01]** A **bundled seed snapshot** of the
  curriculum ships in the app (`assets/curriculum/letters.json` + `lessons.json`, the
  existing files) and is the fallback the app loads on a cold, no-network first run.
- **D-03:** **[HARD CONSTRAINT]** Firestore **offline persistence is enabled**, so after the
  first successful sync the app works fully offline indefinitely. The practice/scoring path
  must never block on a network round-trip.
- **D-04:** `CurriculumRepository` is the single seam to change — it currently loads from
  `rootBundle` (`lib/data/curriculum_repository.dart`). The Firestore read + cache + bundled-
  seed fallback live behind this same interface (`getLetters`/`getLetter`/`getLessons`/
  `getDefaultToleranceRamp`/`getExercises`) so the rest of the app is unaffected.
- **D-05:** The load-time closed-loop/direction/range validator (`validateReferenceStrokes`)
  MUST still run over letters sourced from Firestore — an invalid stroke must never reach the
  scorer regardless of source.

### Firestore data model
- **D-06:** **Doc-per-letter, nested.** One document per letter (doc id = letter `id`, e.g.
  `alif`) in a `letters` collection, with `referenceStrokes`, `commonMistakes`, `tolerances`,
  `audio`, `forms` as nested arrays/maps — mirrors `letters.json` 1:1 for lossless round-trip.
- **D-07:** A `lessons` collection mirrors `lessons.json` (doc per lesson, doc id = lesson
  `id`). The file-level `defaultToleranceRamp` is stored as a single config doc (e.g.
  `meta/toleranceRamp` or a field on a `config` doc) — planner picks the exact location.
- **D-08:** Round-trip parity is a verification gate: a letter read back from Firestore must
  deserialize into the same `Letter` model as the bundled JSON (no field loss).

### Auth & security — FULL foundation, child-safe (owner: "make auth fully, v2-ready")
- **D-09:** **Stand up the full Firebase Auth foundation now**, ready for the rest of the
  project and v2 — but with a HARD child-safety guardrail: **children never log in and never
  get a real account or PII** (Decided: no accounts, minimum child data, no real names). "Full"
  = complete auth *infrastructure*, not a login wall in front of kids.
- **D-09a:** **Enable real providers** in `qalam-app-bd7d0`: **Anonymous + Email/Password +
  Google** (and any others the owner wants) so the infrastructure is fully provisioned.
- **D-09b:** **v1 runtime auth = Anonymous**, signed in on app launch — the identity that reads
  curriculum and will later call the v2 tutor. **No login UI is built in v1** (children don't
  sign in). Zero PII.
- **D-09c:** **Architect anonymous→permanent account linking** (Firebase's `linkWithCredential`
  pattern) so a v1 anonymous identity can be **upgraded** to a real parent/owner account in v2
  **without losing identity or local progress**. This is the key "v2-ready" piece.
- **D-10:** Firestore **security rules**: curriculum (`letters`/`lessons`/config) is **readable
  by any signed-in identity** (`request.auth != null`, incl. anonymous); **all client writes
  denied**. Content writes happen only via the `firebase-admin` service account or the console
  as project owner. Rules are written to be **tightened by custom-claim roles**
  (parent / admin / tutor) in v2 — leave a clear seam.
- **D-10a:** **App Check** is the real anti-abuse lock (attests requests come from the genuine
  app). Deferred to Phase 10 hardening, but the rules/design should not conflict with adding it.
- **D-11:** No child profile, progress, or stroke data is written to Firestore in this phase —
  that data stays in local Drift (per Phase 5/6). Curriculum content only. (When account
  linking lands in v2, any child→cloud sync must re-clear the child-data-minimal bar first.)

### Tooling & authoring path
- **D-12:** **App wiring uses the official FlutterFire CLI** (`flutterfire configure`) —
  generates `firebase_options.dart` + `google-services.json`, adds `firebase_core` /
  `cloud_firestore` / `firebase_auth`. Canonical setup, not a script.
- **D-13:** **Seeding uses a Python `firebase-admin` script** (per CLAUDE.md "Python over
  TypeScript for all backend and tooling"; owner is fluent in Python). It loads alif + the
  lesson skeleton from the existing JSON into Firestore. This is the standard bulk-seed path.
- **D-14:** **Ongoing edits via the Firebase Console.** Doc-per-letter is console-friendly;
  the owner/mother edit content directly there as project owner. No custom authoring tool in
  this phase (a polished in-app authoring UI is deferred to Phase 7+).
- **D-15:** Because the app reads Firestore live, the bundled seed snapshot (D-02) must be
  **refreshable** — provide a way to regenerate `letters.json`/`lessons.json` from Firestore
  (Python `firebase-admin` export) so the shipped offline fallback can be kept current.

### Project & infrastructure
- **D-16:** **Reuse the existing `qalam-app-bd7d0` project** (already authenticated as the
  owner). Use **Cloud Firestore in Native mode**, **Spark (free) tier** for now.
- **D-17:** **Ignore the Data Connect (Cloud SQL/Postgres) config** in the sibling
  `../qalam_ink_spike` folder — that was a separate spike; this phase uses Firestore per the
  Decided stack. Do not enable billing for Data Connect.

### Claude's Discretion
- **Firestore database location/region** is a **permanent** choice — pick a low-latency region
  for the owner's locale at execution (check whether `qalam-app-bd7d0` already has a location
  set first; if so, reuse it). Flag the choice to the owner before creating the database.
- Exact storage location of `defaultToleranceRamp` and any `config`/`meta` doc layout (D-07).
- Whether to gate Firestore reads behind auth-ready state in the repository, and the precise
  cache/seed-fallback ordering, as long as D-02/D-03 hold.
- Android `minSdk`/Gradle bumps required by the Firebase plugins.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `CurriculumRepository`: already abstracts curriculum access behind async getters and an
  in-memory cache with a test-injection constructor (`.fromStrings`). Extend it for the
  Firestore-live + seed-fallback path without touching call sites.
- `assets/curriculum/letters.json` / `lessons.json`: become both the **seed source** (Python
  → Firestore) and the **bundled offline fallback** (D-02).
- `validateReferenceStrokes`: reuse as-is over Firestore-sourced letters (D-05).

### Established Patterns
- Riverpod (codegen, `@Riverpod(keepAlive: true)`) for the repository provider — Riverpod 3
  in this project (see memory: StreamProvider.future hangs; prefer AsyncNotifier bridge for
  live streams if a live Firestore stream is used).
- Drift holds all child/profile/progress data locally — Firestore is curriculum-only (D-11).
- l10n generated files are gitignored — unaffected here, but run `flutter gen-l10n` after
  fresh checkout (existing project convention).

### Integration Points
- App bootstrap (`main.dart`) — Firebase init + anonymous sign-in before the first curriculum
  read; the onboarding/boot gate already does a pre-runApp async read, so there's an existing
  seam for an async Firebase init.
- `pubspec.yaml` — add firebase_core / cloud_firestore / firebase_auth; declare nothing new
  for assets (seed files already declared).

</code_context>
