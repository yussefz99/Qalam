# Phase 06.1 Plan 05: Firestore Security Rules Summary

**The server-enforced access-control half of the curriculum backend: `letters`/`lessons`/`meta` are readable only by a signed-in identity (the anonymous one from Plan 01), every client write is denied (content flows only through the Plan 03 admin-SDK tooling), a deny-by-default catch-all guarantees zero child-PII surface (D-11), and a commented custom-claim seam keeps the v2 role path obvious — authored, wired into firebase.json/.firebaserc, and DEPLOYED live to qalam-app-bd7d0.**

## Performance
- **Duration:** ~10 min
- **Completed:** 2026-06-14
- **Tasks:** 2 (Task 1 auto; Task 2 human-action gate — deploy executed autonomously, Playground checks documented as pending)
- **Files:** 4 (3 created, 1 modified)

## Accomplishments
- **Curriculum locked read-only for clients (D-10, ROADMAP SC#5).** `firestore.rules` (rules_version '2') gives `letters`/`lessons`/`meta` `allow read: if request.auth != null` and `allow write: if false`. A signed-in anonymous client can read curriculum; an unauthenticated client cannot; no client can write — writes happen only via the admin-SDK tooling (Plan 03), which bypasses rules.
- **Deny-by-default catch-all + zero child-PII surface (D-11).** A final `match /{document=**} { allow read, write: if false; }` denies every path not explicitly listed. There is **no** child profile/progress/stroke collection match — child data stays in local Drift, and the catch-all blocks any future or accidental child path.
- **v2 custom-claim seam present (D-10).** Each collection carries a commented `// allow write: if request.auth.token.role == 'admin';` directly above its `allow write: if false;`, so tightening by parent/admin/tutor roles in v2 is a deliberate uncomment, not a rewrite.
- **App-Check-compatible (D-10a).** The rules use no construct that conflicts with adding App Check in Phase 10 — App Check can be layered on without touching this file.
- **Deploy targets wired without clobbering FlutterFire config (D-17).** `firebase.json` now declares `{"firestore":{"rules":"firestore.rules"}, "flutter": {...}}` — the firestore target merged in, the generated flutter platform/dart blocks preserved. `.firebaserc` defaults the project to `qalam-app-bd7d0` (D-16).
- **Rules DEPLOYED live.** `firebase deploy --only firestore:rules --project qalam-app-bd7d0` compiled `firestore.rules` successfully and **released** it to cloud.firestore (exit 0, "Deploy complete!"). The access-control rules are now live on the production database.
- **Repeatable verification checklist authored.** `test/firestore/rules.test.md` documents the five server-side Rules Playground checks (authed read ALLOWED, unauthed read DENIED, authed write DENIED, catch-all path DENIED, no child-data surface) with exact steps and expected outcomes, plus an emulator alternative.

## Decisions Made
- **Merged into firebase.json, did not clobber it.** The flutterfire-generated `firebase.json` carried the flutter platform/dart config; the firestore target was prepended without disturbing it (verified: `firebase_options.dart` still present, valid JSON). No Data Connect / billing enabled (D-17).
- **Custom-claim seam stays commented.** v1 denies all client writes outright; the role guard is documented inline so v2 uncomments it deliberately once `parent/admin/tutor` claims exist — no live role logic before roles do (D-10).
- **No child-data match by design.** Rather than author a child collection and lock it down, there is simply no child match — the catch-all denies it. This makes the D-11 "no child PII surface" guarantee structural, not a rule that could be loosened.
