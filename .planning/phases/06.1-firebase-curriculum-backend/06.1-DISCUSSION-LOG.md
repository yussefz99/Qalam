## Runtime data source (offline contract)

| Option | Description | Selected |
|--------|-------------|----------|
| Bundle-only | App reads bundled letters.json at runtime; Firestore author-only + export to bundle. Guarantees offline, zero runtime network. (Claude recommended) | |
| Live Firestore + cache | App reads curriculum live from Firestore with offline persistence + bundled seed for first run. Content updates without app release; adds runtime cloud dependency. | ✓ |

**User's choice:** Live Firestore + cache.
**Notes:** Claude flagged the offline implication — a fresh, no-network install has no curriculum until first sync. Captured as a HARD CONSTRAINT (CONTEXT D-02/D-03): ship a bundled seed snapshot as cold-start fallback AND enable Firestore offline persistence, so PLAT-01 (Phase 10) still holds.

## Auth & security

| Option | Description | Selected |
|--------|-------------|----------|
| Admin-SDK writes, client locked, no end-user Auth | Writes via service account only; rules deny client access; no Firebase Auth in v1. (Claude recommended) | |
| Add Firebase Auth now | Stand up Firebase Auth this phase to gate reads/writes; groundwork for v2 tutor. | ✓ |

**User's choice:** Add Firebase Auth now. Auth method deferred to Claude ("idk what do you think… set it up properly").
**Notes:** Claude decided **anonymous auth** as the proper fit: every app instance gets an authenticated identity (so rules can require `request.auth != null` for reads) with zero PII and no accounts — consistent with the child-data-minimal / no-account Decided constraints. Content writes remain admin-only via the service account / console (CONTEXT D-09/D-10).

## Tooling & authoring path

| Option | Description | Selected |
|--------|-------------|----------|
| Firebase Console + Python seed | Python firebase-admin seeds alif + lessons once; ongoing edits in the console. (Recommended) | ✓ |
| Python seed/export scripts only | Seed + snapshot export; edits by re-running scripts from source JSON. | |
| Custom in-app/admin authoring tool | Polished authoring UI; real scope, deferred. | |

**User's choice:** Firebase Console + Python seed.
**Notes:** User questioned "why python scripts — set it up correctly." Clarified the two layers: app wiring uses the official **FlutterFire CLI** (`flutterfire configure`) — the canonical setup, not a script; only the data seed uses a script, and Python `firebase-admin` is mandated by CLAUDE.md ("Python over TypeScript for all tooling") and is the standard bulk-seed path (CONTEXT D-12/D-13/D-14). A refreshable snapshot export keeps the bundled offline fallback current (D-15).
