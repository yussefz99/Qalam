## Summary

This phase wires the existing Qalam Flutter app to Cloud Firestore (in the already-created `qalam-app-bd7d0` project) as the live cloud source-of-truth for curriculum, while preserving the hard offline-first promise (PLAT-01) through Firestore's built-in offline persistence plus a bundled JSON seed fallback. The work decomposes into five independent tracks: (1) FlutterFire CLI wiring + Android Gradle/minSdk bumps, (2) extending `CurriculumRepository` to a Firestore→bundled-fallback read path behind its unchanged async-getter API, (3) a full v2-ready Firebase Auth foundation (anonymous-at-launch, providers enabled, account-linking architected), (4) locked Firestore security rules with a custom-claim seam, and (5) Python `firebase-admin` seed + export scripts.

The single biggest technical landmine is the **Firestore nested-array constraint**: `StrokeSpec.points` is `List<List<double>>` (an array of `[x, y]` pairs), and **Firestore does not allow an array element to itself be an array**. The seed script and the Firestore deserializer must transform each point pair into a map (`{x, y}`) on write and back to `[x, y]` on read. This transform is the crux of D-06/D-08 round-trip parity and must be the spine of the model-mapping task.

The second risk is Riverpod-3 stream behavior (project memory: Riverpod 3 pauses unlistened `StreamProvider`s and a bare `.future` read hangs). Because curriculum is **near-static** and the read happens once at boot into a kept-alive cache, the cleanest path is a **one-shot `.get()` read** (not `.snapshots()`), which sidesteps the stream-pause class of bugs entirely. Live `.snapshots()` is unnecessary for content that changes only via console edits between releases; if live updates are ever wanted, use the project's existing `_bindDriftStream`-style AsyncNotifier bridge — but this phase does not need it.

**Primary recommendation:** FlutterFire CLI wires the app; `CurriculumRepository` gains a `FirebaseFirestore` dependency and does a one-shot `.get()` (Firestore serves from cache when offline) → on any failure/empty falls back to the existing `rootBundle` JSON load → then runs `validateReferenceStrokes` over whichever source won → caches the validated result. Points are stored as `{x, y}` maps in Firestore. Anonymous sign-in completes in the existing pre-`runApp` boot gate before the first read. Seed/export via Python `firebase-admin`. Test the repository with `fake_cloud_firestore` (no emulator, no network).

### Pattern 1: Firestore-first read with bundle fallback (D-01/D-02/D-04/D-05)
**What:** Extend `_ensureLoaded()` to try Firestore one-shot `.get()`, fall back to `rootBundle`, validate the winner, cache.
**When to use:** The core repository change. Preserves the existing async-getter API and `.fromStrings` test seam.
**Example:**
```dart
// Source: composed from existing curriculum_repository.dart + FlutterFire .get() pattern
// (firebase.flutter.dev/docs/firestore/usage). Test seam (_lettersJsonOverride) wins first
// so .fromStrings tests stay network-free.
Future<void> _ensureLoaded() async {
  if (_letters != null) return;

  List<Letter> parsed;
  if (_lettersJsonOverride != null) {
    parsed = _parseLettersJson(_lettersJsonOverride!);   // existing test path
  } else {
    parsed = await _loadFromFirestoreOrBundle();          // NEW
  }
  parsed.sort((a, b) => a.introOrder.compareTo(b.introOrder));

  // D-05 guard — UNCHANGED, runs on whichever source won. Validate into a local
  // so a throw does not poison the cache.
  for (final letter in parsed) {
    final v = validateReferenceStrokes(letter.referenceStrokes);
    if (v.isNotEmpty) {
      throw StateError('Invalid referenceStrokes for "${letter.id}": ${v.join('; ')}');
    }
  }
  _letters = parsed;
  _lettersView = List.unmodifiable(_letters!);
  // ...lessons + defaultToleranceRamp identically (Firestore → bundle fallback)...
}

Future<List<Letter>> _loadFromFirestoreOrBundle() async {
  try {
    final snap = await _firestore.collection('letters').get();   // cache-served offline
    if (snap.docs.isNotEmpty) {
      return snap.docs.map((d) => _letterFromFirestore(d.data())).toList();
    }
  } catch (_) {
    // network/permission/cold-first-run → fall through to bundle (D-02)
  }
  final raw = await rootBundle.loadString('assets/curriculum/letters.json');
  return _parseLettersJson(raw);
}
```

### Pattern 2: Point codec — the nested-array workaround (D-06/D-08)
**What:** Convert each `[x, y]` point to a `{x, y}` map on write (Python) and back on read (Dart), because Firestore forbids arrays-of-arrays.
**When to use:** Everywhere `StrokeSpec.points` crosses the Firestore boundary — seed script, export script, and the Dart deserializer.
**Example:**
```dart
// Source: derived from firebase.flutter.dev data-types + the nested-array constraint
// (github.com/FirebaseExtended/flutterfire#1637). Firestore stores points as
// [{x:0.5,y:0.191}, ...]; rebuild the List<List<double>> the model expects.
StrokeSpec _strokeFromFirestore(Map<String, dynamic> s) {
  final rawPoints = (s['points'] as List).cast<Map<String, dynamic>>();
  final points = rawPoints
      .map((p) => [(p['x'] as num).toDouble(), (p['y'] as num).toDouble()])
      .toList();
  return StrokeSpec(
    order: s['order'] as int,
    label: s['label'] as String,
    type: s['type'] as String? ?? 'line',
    points: points,
    direction: s['direction'] as String,
  );
}
```
```python
# Source: firebase-admin Python; seed-side encode
def encode_points(pairs):  # [[0.5, 0.191], ...] -> [{"x":0.5,"y":0.191}, ...]
    return [{"x": float(p[0]), "y": float(p[1])} for p in pairs]
```

### Anti-Patterns to Avoid
- **Using `.snapshots()` for curriculum.** It introduces Riverpod-3 stream-pause/`.future`-hang risk for data that changes only between releases. Use one-shot `.get()`.
- **Blocking the practice/scoring path on a Firestore round-trip.** Reads must hit cache or fall back to bundle; never `await` a live network call on the trace path (D-03).
- **Storing points as nested arrays.** Firestore rejects arrays-of-arrays — the write will throw. Always use `{x, y}` maps.
- **Hard-picking mismatched Firebase plugin majors.** Let `flutterfire configure` / `flutter pub add` resolve the compatible set.
- **Committing the service-account JSON key.** Add it to `.gitignore`; reference via `GOOGLE_APPLICATION_CREDENTIALS` env var.
- **Building a login UI for children.** D-09b: no child login, no PII. Anonymous only at runtime.
- **Letting an invalid Firestore stroke bypass the validator.** D-05: `validateReferenceStrokes` runs on the Firestore source too, not just the bundle.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Offline cache of curriculum | Custom SQLite/file cache of fetched docs | Firestore offline persistence (on by default, Android) | Firestore handles cache, eviction, cache-served reads automatically (D-03) |
| First-run-no-network floor | Custom "have I synced yet?" flag | Bundled JSON fallback under the existing `rootBundle` loader (D-02) | The loader already exists; just layer it under the Firestore try |
| Anonymous identity | Custom device-id scheme | `firebase_auth.signInAnonymously()` | Gives a real auth identity that rules gate on and that `linkWithCredential` can upgrade (D-09c) |
| Access control | Client-side "is this allowed?" checks | Firestore security rules (server-enforced) | Client checks are bypassable; rules enforce read-auth + write-deny server-side (D-10) |
| Bulk content write | Loop of single `set()` calls in Dart | Python `firebase-admin` batch/bulk writer | Admin SDK bypasses rules, runs off-device, idempotent by doc-id (D-13) |
| Firestore test double | Real network / live project in tests | `fake_cloud_firestore` | In-memory, deterministic, no network, no emulator process |

**Key insight:** The offline-first requirement (PLAT-01) is satisfied by **composition of two built-ins** — Firestore persistence (warm offline) + the existing bundle loader (cold first run) — not by any new caching machinery. The phase's net-new code is small: a point codec, a Firestore→model mapper, a fallback branch, an anonymous sign-in call, a rules file, and two Python scripts.

## Common Pitfalls

### Pitfall 1: Firestore nested-array rejection (THE landmine)
**What goes wrong:** Writing `StrokeSpec.points` (`List<List<double>>`) directly to Firestore throws `Invalid data. Nested arrays are not supported`.
**Why it happens:** Firestore arrays cannot contain array elements. `points` is exactly an array of `[x,y]` arrays.
**How to avoid:** Encode each pair as a `{x, y}` map (Pattern 2). Apply identically in the Python seed (`encode_points`), the Python export (`decode_points`), and the Dart deserializer. Round-trip parity (D-08) is the verification gate.
**Warning signs:** Seed script throws on the first letter with strokes (alif has one stroke, 64 points); or a read returns points as `List<Map>` and `StrokeSpec.fromJson` casts blow up.

### Pitfall 2: `.future` read of a StreamProvider hangs (project memory)
**What goes wrong:** Riverpod 3 pauses unlistened `StreamProvider`s; a bare `.future` read hangs forever.
**Why it happens:** Riverpod-3 lifecycle change.
**How to avoid:** Do not model curriculum as a `StreamProvider` at all — use a one-shot `.get()` inside the existing kept-alive `curriculumRepositoryProvider`. If a future phase wants live updates, use the project's `_bindDriftStream`-style AsyncNotifier bridge, not a bare StreamProvider.
**Warning signs:** App hangs on first curriculum load with no error.

### Pitfall 3: minSdk too low for Firebase plugins
**What goes wrong:** Firebase plugins require `minSdkVersion >= 23`; build fails or crashes if lower. [CITED: firebase.flutter.dev/docs/manual-installation/android]
**Why it happens:** Firebase Android SDK floor.
**How to avoid:** The project uses `minSdk = flutter.minSdkVersion` (Flutter-managed). Flutter 3.41's default minSdk is already 24, so this is **likely already satisfied** [ASSUMED — verify with `flutter build apk` or by checking the resolved value]. If the build complains, set `minSdk = 23` (or higher) explicitly in `android/app/build.gradle.kts`. minSdk >= 21 means multidex is on by default — no multidex library needed.
**Warning signs:** Gradle error referencing `minSdkVersion`, or a 64K-method dex-merge error (→ add `multiDexEnabled true`, but unlikely at minSdk 24).

### Pitfall 4: `google-services` Gradle plugin not applied
**What goes wrong:** `flutterfire configure` generates `google-services.json` but the build can't find Firebase config → runtime init failure.
**Why it happens:** The `com.google.gms.google-services` Gradle plugin must be declared (settings.gradle.kts) and applied (app/build.gradle.kts). Recent `flutterfire configure` automates this, but verify.
**How to avoid:** After `flutterfire configure`, confirm `id("com.google.gms.google-services")` appears in `android/settings.gradle.kts` (as `apply false`) and is applied in `android/app/build.gradle.kts`. Run `flutter build apk --debug` once to prove it links. [CITED: firebase.google.com/docs/flutter/setup]
**Warning signs:** `[core/no-app]` or "Default FirebaseApp is not initialized" at launch.

### Pitfall 5: `defaultToleranceRamp` lost in round-trip
**What goes wrong:** The file-level `defaultToleranceRamp` (and per-lesson `toleranceRamp`) lives outside any lesson doc; if the seed script doesn't place it in a config/meta doc, the repo's `getDefaultToleranceRamp()` silently returns the hard-coded default (`['loose','normal','strict']`) — a quiet behavior change masked by the defensive parse.
**Why it happens:** `lessons.json` is `{defaultToleranceRamp, lessons[]}` — two shapes in one file. Firestore has no "file."
**How to avoid:** Per D-07, store `defaultToleranceRamp` as a field on a `meta` doc (e.g. `meta/toleranceRamp`) and have the repo read it before falling back to the default. Round-trip test must assert the ramp survives.
**Warning signs:** Tolerance ramp tests pass against the bundle but the Firestore path always returns the default.

### Pitfall 6: `signedOff: false` / empty-stroke letters and the validator
**What goes wrong:** Only alif is `signedOff: true` with strokes; the other 27 carry `referenceStrokes: []` + `signedOff: false`. The seed writes all 28; the validator must not choke on empty strokes (it currently doesn't — empty list yields no violations), but the round-trip must preserve `signedOff` and empty `referenceStrokes` exactly.
**Why it happens:** Partial curriculum (alif signed, rest skeleton).
**How to avoid:** Round-trip test must include at least one empty-stroke letter and assert `signedOff`/`mistakesStatus`/empty `referenceStrokes` survive. Confirm the validator is a no-op on `[]`.
**Warning signs:** A skeleton letter comes back from Firestore with `signedOff` defaulted or strokes coerced.

## Security Domain

> security_enforcement = true, ASVS level 1 → required.

### Applicable ASVS Categories
| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | firebase_auth anonymous sign-in; providers provisioned (Anonymous/Email/Google). No passwords stored client-side. |
| V3 Session Management | yes | Firebase-managed ID tokens; anonymous session persists on-device; `linkWithCredential` upgrades without new session. |
| V4 Access Control | yes | **Firestore security rules** — read requires `request.auth != null`; all client writes denied; custom-claim seam for v2 roles. Server-enforced. |
| V5 Input Validation | yes (read-side) | `validateReferenceStrokes` over Firestore-sourced data (D-05); defensive parse on optional fields (existing model pattern). |
| V6 Cryptography | no (this phase) | No custom crypto. Firebase handles transport (TLS) + token signing. Service-account key handling is V8/secret-mgmt, not crypto. |

### Known Threat Patterns for FlutterFire + Firestore
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Unauthorized read of curriculum / future PII | Information Disclosure | Rules require `request.auth != null`; D-11 keeps child data OUT of Firestore entirely this phase |
| Malicious client writes / content tampering | Tampering | Rules: client `write: false`; writes only via admin SDK service account |
| Spoofed app hitting Firestore (no App Check) | Spoofing | App Check deferred to Phase 10 (D-10a) — rules must not conflict with adding it; until then anonymous-auth + write-deny limit blast radius |
| Service-account key leak (seed scripts) | Elevation of Privilege | Key in `.gitignore`, referenced via env var, never committed; least-privilege SA |
| Child PII accidentally synced | Information Disclosure | D-11 hard rule: no child profile/progress in Firestore; rules have no child-data collection to write |

**Recommended rules skeleton (D-10):**
```javascript
// Source: firebase.google.com/docs/firestore/security/get-started (structure)
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Curriculum: readable by any signed-in identity (incl. anonymous); no client writes.
    match /letters/{letterId}  { allow read: if request.auth != null; allow write: if false; }
    match /lessons/{lessonId}  { allow read: if request.auth != null; allow write: if false; }
    match /meta/{doc}          { allow read: if request.auth != null; allow write: if false; }
    // v2 seam (commented): tighten by custom claims, e.g.
    //   allow write: if request.auth.token.role == 'admin';
    match /{document=**}       { allow read, write: if false; }   // deny by default
  }
}
```
Deploy: `firebase deploy --only firestore:rules` (rules in `firestore.rules`, target declared in `firebase.json`).
