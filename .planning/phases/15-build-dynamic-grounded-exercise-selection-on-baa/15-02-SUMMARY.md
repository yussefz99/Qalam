## Decisions Made

- **Graph-rail activation predicate (the load-bearing call).** The literal plan text says "G6 rejects an exercise whose prerequisites are uncleared." Applied unconditionally, that rejects every `positionalForms`+ exercise whenever `clearedCompetencies` is empty. Because Pydantic defaults the two new fields to `[]` and `main.py` does `facts = facts_in.model_dump()`, the LIVE endpoint ALWAYS carries the keys as `[]` — so a key-presence check could not distinguish a fresh child, and an unconditional G6 would degrade EVERY online plan run to the AuthoredFallback floor before 15-04 ships the Dart cleared-state (a deployed regression). Resolved by activating the rail only on a KNOWN graph position (`clearedTiers` OR `clearedCompetencies` non-empty). This keeps all four `test_plan_graph.py` cases green (each provides non-empty cleared-state), keeps the deployed endpoint/routing tests green (empty cleared-state → no-op), and is correct: a fresh child at the graph root is pre-graph; G4 + G3 still hold unconditionally.
- **Derived copy keeps top-level fields verbatim, filters only `nodes`.** The server rails just baa this phase; competencies/tiers/_meta are copied as-is so the helpers reason over the same competency-prereq + tier-ladder definitions the asset declares.
- **Fail-closed at import.** A missing/unreadable/malformed graph degrades to an empty graph (never raises) so `/coach` import never 500s (T-15-02-T).

This honors the threat-model intent (gate illegal forward jumps, not legitimate selections lacking resume input — Pitfall 3) without dead-ending a fresh child.

The forward direction is the trap: when 15-04 ships the Dart fields, the server MUST already be re-deployed with these fields, or the live `/coach` 422s the enlarged payload.
