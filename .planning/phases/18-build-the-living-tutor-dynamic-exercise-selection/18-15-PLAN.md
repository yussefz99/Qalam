<objective>
Close the resume-lost-on-relaunch gap (UAT test 7). The debug confirmed this is NOT the nightly job and NOT the already-filed per-child keying gap — it is a Phase-18-07 regression. The durable `LetterGraphPosition.currentExerciseId` IS persisted and read back correctly, but the session-scoped render state that decides WHAT renders (`LetterUnitState.selectionActive` + the shell-local `_presentedId`) is never persisted or re-seeded from it. `start()` resets `selectionActive` to `false` on every fresh call, and a cold relaunch makes a new screen state with `_presentedId == null`, so the render condition `state.selectionActive && _presentedId != null` is always false after boot — the shell falls back to the legacy `_section(index)` walk, whose own `_sectionHintFor` heuristic (keyed on `clearedCompetencies.length`, not on `currentExerciseId`) commonly resolves to section 0 in a short session. That is the "started from scratch" report.

Purpose: A returning child resumes exactly where they were within the unit; the durable cursor already exists — it just needs to drive the render after a cold boot.

Output: `start()` restores `selectionActive: true` when the saved `currentExerciseId` is a real graph node, and the screen's render path uses that restored cursor when `_presentedId` is still null — re-entering presenter mode on the exact node, bypassing the legacy section-hint walk. No Drift schema change (the cursor is already persisted; only the render state needs re-seeding).
</objective>
