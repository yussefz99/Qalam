<objective>
Produce the curriculum source data — the Python font-extraction tool, the full 28-letter
`letters.json`, and the minimal `lessons.json` skeleton — and add the `assets/curriculum/`
declaration to pubspec.yaml so Flutter can load both files via `rootBundle`.

The extraction script reads the bundled NotoNaskhArabic-Regular.ttf, isolates each
letter's discrete contour components as normalized polylines, and writes them to
`tools/candidate_paths.json` for the owner to inspect. The owner then maps each contour
to the corresponding teaching stroke (per D-01, D-02, D-03) and records the result as
`referenceStrokes` in `letters.json`. Alif must have real paths and `signedOff: true`
before this plan is complete (D-08, D-12). All 28 letter entries are authored in this
plan (D-05, D-06): the 27 non-alif letters carry correct structural data (forms,
introOrder, cleanRepsToAdvance) but have `referenceStrokes: []`, `signedOff: false`,
and `mistakesStatus: "placeholder"` (D-07).

Purpose: Establish the single source of truth for curriculum data that Plan 02-02 models
in Dart and Plan 02-03 loads at runtime. Phase 3 consumes alif's reference paths for
the first end-to-end trace.

Output: `tools/extract_reference_paths.py`, `tools/candidate_paths.json` (generated),
`assets/curriculum/letters.json` (28 letters), `assets/curriculum/lessons.json`
(lesson_01), and the pubspec.yaml asset declaration.
</objective>

    Create `tools/extract_reference_paths.py`. The script is a standalone developer tool
    (not imported by the app). It requires fonttools (`pip install fonttools`). It must:

    1. Open `assets/fonts/NotoNaskhArabic-Regular.ttf` using fonttools TTFont. If the
       file is missing, exit with a clear error message showing the expected path.

    2. For each of the 28 Arabic letters by Unicode codepoint (use the cmap table to
       map codepoint to glyph name), extract the glyph from the `glyf` table. The 28
       codepoints are: U+0627 (alif), U+0628 (baa), U+062A (taa), U+062B (thaa),
       U+062C (jeem), U+062D (haa_c), U+062E (khaa), U+062F (daal), U+0630 (dhaal),
       U+0631 (raa), U+0632 (zaay), U+0633 (seen), U+0634 (sheen), U+0635 (saad),
       U+0636 (daad), U+0637 (taa_h), U+0638 (zhaa), U+0639 (ayn), U+063A (ghayn),
       U+0641 (faa), U+0642 (qaaf), U+0643 (kaaf), U+0644 (laam), U+0645 (meem),
       U+0646 (noon), U+0647 (haa_f), U+0648 (waaw), U+064A (yaa).

    3. For each glyph, decompose into discrete contour components. For composite glyphs
       (component references), resolve each component to its own glyph contours. Use
       the glyph's `numberOfContours` to determine if it is simple or composite. For
       simple glyphs, split points into contours using `endPtsOfContours`. For composite
       glyphs, iterate `components` and resolve each recursively to its base glyph's
       contours.

    4. For each contour, convert the TrueType quadratic bezier points to a polyline:
       use uniform resampling at N=64 points. Walk the contour's on-curve and off-curve
       points, interpolate quadratic bezier segments at uniform t parameter spacing
       (subdivide each segment into enough steps to reach 64 total points per contour).

    5. Compute the per-letter bounding box across ALL of that letter's contours combined
       (xMin, yMin, xMax, yMax). Normalize every point in every contour to 0..1 using
       this shared bounding box: `x_norm = (x - xMin) / (xMax - xMin)`, same for y.
       This is the "per-letter 0..1 bounding box" requirement (D-02). Handle degenerate
       bounding boxes (width or height == 0) by clamping to 0.0.

    6. Write output to `tools/candidate_paths.json` in this shape:
       ```
       {
         "alif": {
           "char": "ا",
           "codepoint": "U+0627",
           "contour_count": 1,
           "contours": [
             {
               "index": 0,
               "label": "contour_0",
               "point_count": 64,
               "points": [[x, y], ...]
             }
           ]
         },
         ...
       }
       ```
       Use the letter names from the 28-letter list (alif, baa, taa, etc.) as keys.
       Round all coordinate values to 4 decimal places to keep the file inspectable.

    7. Print a human-readable summary to stdout after writing: for each letter, print
       the letter name, Unicode char, and contour count. Example:
       `alif (ا): 1 contour(s)` — one line per letter, 28 lines total.

    Do not make the script take CLI arguments — it always reads from
    `assets/fonts/NotoNaskhArabic-Regular.ttf` and writes to `tools/candidate_paths.json`
    relative to the project root. Add a top-of-file comment explaining what the script does
    and how to use it (run from the project root with `python tools/extract_reference_paths.py`).

    Then edit pubspec.yaml: in the `flutter:` → `assets:` section, add

    `assets/curriculum/` directory will be created in Task 2 when the JSON files are
    written; Flutter does not require the directory to exist at pubspec edit time.

    NOTE: Do NOT create `tools/candidate_paths.json` manually. It is generated by
    running the script. Task 2 creates `letters.json` using the script's output.
  </action>
