# Phase 02.1.1: Presentation Demo Screens (alif core loop) - Context

**Gathered:** 2026-06-01

<domain>
## Phase Boundary

This phase exists for a **course-staff presentation on 2026-06-02**. The deliverable is a
set of **beautiful, screenshot-ready, navigable Flutter screens** that walk through the full
Qalam core loop for **alif** — Home → Watch → Trace → Tutor Feedback → Celebration — built
with the **real Flutter widgets + design system** but fed **mocked/static content**. The app
is NOT run live in front of staff; the presenter shows the screens (and/or screenshots) and
narrates. The tutor's warm, specific feedback is **depicted in copy**, not produced by a
model or scorer.

**In scope:**
- Six screen states, navigable in order (tappable walkthrough, no dead ends):
  1. **Home** — de-gamified: mascot, warm static greeting, alif "Today's lesson" card.
  2. **Watch** — dotted alif with the numbered stroke-order guide + mascot demonstrating.
  3. **Trace** — a half-traced letter with live ink rendered over the dotted guide.
  4. **Feedback · miss (HERO)** — failing stroke highlighted in coral + a specific named
     fix in the tutor's voice.
  5. **Feedback · clean pass** — warm, specific praise ("Beautiful — straight and tall.
     أحسنت.").
  6. **Celebration** — mascot + exactly ONE quiet star + a warm line + Back home.
- Bundling the real brand **mascot + icon SVGs** into the app `assets/` (they live only in
  the design kit today) so the screens render the real illustrations.
- Tokens-only, RTL, tablet-first, gen-l10n copy — pixel-faithful to the design.

**Out of scope (deliberately mocked / deferred to Phase 3):**
- The geometric **stroke scorer** — no real scoring; the "miss" and "pass" states are
  hardcoded demo states.
- **Stylus capture logic / palm rejection** — the "live ink" can be a static painted path
  or a simple non-scored capture; no per-stroke order/count engine.
- **Drift persistence**, clean-rep counting, mastery recording — none; the star is depicted.
- **ML Kit**, any **network**, any **model/API** — nothing leaves the device.

**Relationship to Phase 3:** These are the **UI layer of Phase 3 built first with mocked
data**. The same widgets become Phase 3's real screens once the engine (scorer, capture,
persistence) is wired behind them — so the work is reusable, not throwaway. (This phase
therefore overlaps with, and supersedes for now, the demo-home plan 03-05 and the asset-
bundling plan 03-00 that were tentatively added to Phase 3 — see Deferred.)
</domain>

<decisions>
## Implementation Decisions

### Nature of the build
- **DP-01: Mocked/static content, zero engine.** No scorer, no capture engine, no Drift, no
  ML Kit, no network, no model/API. Every screen renders a hardcoded "happy path" state.
  This is what makes it fast and impossible to break on stage.
- **DP-02: Real widgets + real design system, so it's not throwaway.** Build with the actual
  Flutter widget structure, `QalamColors`/`QalamSpace`/`QalamTextStyles`/`QalamFonts` tokens,
  `ArabicText` RTL island, and gen-l10n copy — so Phase 3 can wire the real engine behind
  these exact widgets later. No raw hex, no magic numbers, no inline fonts.
- **DP-03: Navigable walkthrough.** The five steps are tappable in order
  (Home → Watch → Trace → Feedback → Celebration → Home) via go_router, no dead ends — so the
  presenter can click through live if asked, and screenshots come from real navigation.

### The screens & the tutor voice
- **DP-04: All six states ship** (Home, Watch, Trace, Feedback·miss, Feedback·pass,
  Celebration) — the full narrative the presenter walks through.
- **DP-05: Tutor voice is depicted in copy, and it must be specific.** The miss state names
  the exact fix in a warm teacher's voice (e.g. "Start your alif at the top and pull straight
  down — yours leans left"); the pass state gives specific praise. **Never** a generic "Oops,
  try again." Source the mistake wording from alif's `commonMistakes[].feedback` in
  `assets/curriculum/letters.json` (Phase 2 data) for authenticity where practical.
- **DP-06: Anti-gamification still binding.** Even in the demo: NO running star counter, NO
  weekly tally, NO streaks/badges/confetti/timers, NO per-lesson 3-star rating. Celebration
  shows exactly ONE quiet star. Honors the Decided rules and [[design-predates-antigamification]].

### Fidelity & assets
- **DP-07: Reuse Phase 3's design contract.** `03-UI-SPEC.md` and `docs/design/kit/` are the
  canonical visual source. Build to the mockups (`home.png`, `02-flow.png`, `03-flow.png`,
  `05-celebration-final.png`) with the gamification chrome omitted per DP-06. Mockups show
  baa; this ships **alif**.
- **DP-08: Bundle the real brand assets in this phase.** Copy the 5 mascot poses → `assets/
  mascot/` and the 5 icons → `assets/icons/` from `docs/design/kit/project/assets/`, register
  `assets/mascot/` in `pubspec.yaml`. Pose→state: idle=home, write=Watch demo, cheer=
  celebration, try-again=miss feedback. (Today only a text-wordmark `logo.svg` is bundled —
  see [[brand-assets-not-bundled]].) Keep every `SvgPicture.asset` behind a graceful
  missing-asset fallback.
- **DP-09: Screenshot-ready.** Clean tablet RTL states, no debug banners/overlays; each
  screen defaults to its "hero" mocked state so a screenshot is presentation-clean.

### Honesty framing (for the presenter, not a build constraint)
- **DP-10:** Present these as "**screens implemented; the scoring engine is the next phase**."
  Accurate and a strong milestone position — do not imply the scorer/persistence are live.

### Claude's Discretion
- How "live ink" on the Trace screen is faked (a static painted path, a captured-but-unscored
  `Listener` stroke, or a short pre-baked animation) — pick whatever looks best and is safe.
- Whether the navigable walkthrough reuses the real `/practice` route or a dedicated demo
  route group; how mocked state is injected (Riverpod overrides, constructor params, or simple
  enum-driven screen state).
- Exact widget decomposition, as long as it matches the design system and is reusable by
  Phase 3.
</decisions>

<specifics>
## Specific Ideas

- The **Feedback · miss screen is the hero shot** — it proves the app diagnoses a *specific*
  stroke problem and gives a *specific* fix. That lands harder with educators than a generic
  AI "good job." Make it the most polished screen.
- Greeting name is a **static placeholder** (no profile system — that's Phase 5).
- Use Western numerals; Arabic content via Noto Naskh / Cairo, never bold/italic/letterSpaced.
</specifics>
