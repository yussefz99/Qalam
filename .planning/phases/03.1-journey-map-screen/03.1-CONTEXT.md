# Phase 03.1: Journey Map Screen — Context

**Gathered:** 2026-06-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement the Journey Map screen in Flutter: a winding-path progress view showing all
28 letters as nodes across 4 rows of 7, with 4 node states, pulse animation on the
current letter, a Level 1 Quiz checkpoint, and a locked Level 2 banner. The screen is
unlocked from the Home nav-rail and navigates to the Practice screen on node tap.

This is an inserted phase between Phase 3 (Trace One Letter) and Phase 4 (Scoring
Quality). It uses mocked progress data (no real DB wiring until Phase 6) and the
existing design system + CustomPainter for the snake path.

</domain>

<decisions>
## Implementation Decisions

### Node Interaction
- **D-01:** ALL nodes are tappable, not just the current node.
- **D-02:** Tapping the **current** (teal) node → navigates to `/practice` for that letter.
- **D-03:** Tapping a **complete** (green) node → also navigates to `/practice` for that letter (replay). Same route, same screen.
- **D-04:** Tapping **future** or **locked** nodes → inert (no navigation, no feedback). Visual distinction is sufficient.

### Data Source
- **D-05:** Use **mocked progress data** for now — a static snapshot baked into a mock provider. No real `ProgressRepository` wiring yet.
- **D-06:** The mock simulates: first 3 letters (Alif, Baa, Taa) = complete; letter 4 (Thaa) = current; letters 5–28 = future. This matches the journey_preview.html design demo state.
- **D-07:** The "current" letter is computed as the first non-mastered letter in curriculum order — so when the mock is later replaced with real data, the logic is already correct.
- **D-08:** Real wiring (Phase 6) will swap the mock provider for the live `ProgressRepository` + `CurriculumRepository` — the screen itself doesn't change, only the provider.

### Winding Path Layout
- **D-09:** 7 nodes per row × 4 rows = 28 letters. All fit on one screen — no vertical scrolling.
- **D-10:** Row direction alternates: Row 1 R→L (Alif at right), Row 2 L→R, Row 3 R→L, Row 4 L→R. Starting from the right feels natural for Arabic.
- **D-11:** Path rendered via `CustomPainter` with smooth bezier corners (Q curves, ~40px radius). Colour: grey (#D6E8E8) for the full trail; a green overlay segment for completed letters.
- **D-12:** Node sizes: 68px circles (slightly smaller than home lesson card to fit 7 per row comfortably).

### Node Visual States
- **D-13:** `complete` → leaf green fill, 5px drop-shadow, gold ★★★ badge top-right.
- **D-14:** `current` → ink-teal fill, pulsing glow animation (2.4s ease-in-out, AnimationController).
- **D-15:** `future` → white fill, dashed aqua-edge border, muted label.
- **D-16:** `locked` → reserved for edge cases; not expected in Phase 3 state (all 28 letters exist in curriculum). Use warm-grey fill if encountered.

### "Today" label
- **D-17:** Current node shows a "TODAY" pill chip above the circle (ink-teal background, white text, uppercase, 10pt). Positioned with a Stack widget above the circle.

### Level Structure
- **D-18:** Level 1 header pill centered at top: "Level 1 · LETTERS · 3 of 28 mastered".
- **D-19:** Level 1 Quiz checkpoint below the last letter (at bottom center): a rounded-rect with lock icon, "Level 1 Quiz", "Complete all 28 letters to unlock".
- **D-20:** Level 2 banner below checkpoint: pill/card with 🔒, "Level 2 — Words", "Unlocks after completing the Level 1 Quiz". Opacity 0.72, dashed border.

### Navigation & Routing
- **D-21:** Add `/journey` route to `app_router.dart`. The Journey nav item in `home_screen.dart` is unlocked: `onTap: () => context.go('/journey')`.
- **D-22:** The "See journey" ghost button in `mastery_celebration.dart` (Phase 3) is wired to `/journey` in this phase (it was a no-op placeholder).

### Anti-gamification
- **D-23:** No running star counter anywhere on the Journey screen. Stars appear only as mastery badges on completed nodes.
- **D-24:** No "THIS WEEK" tally, no streak, no "+N keep going" copy. The level progress ("3 of 28 mastered") is factual progress information, not a score to chase.

</decisions>

<specifics>
## Specific Ideas

- **Reference implementation:** `docs/design/kit/project/ui_kits/qalam_app/journey_preview.html` is the pixel-faithful reference. The planner should use it as the spec, not re-derive layout from scratch.
- **Path layout values (from preview):** X positions for 7 nodes across 1180px content: 85, 254, 422, 590, 758, 926, 1094. Row Y positions: 160, 290, 420, 550. Checkpoint at 590,660. Level 2 banner at 590,752.
- **Path SVG → CustomPainter translation:** The winding path uses cubic bezier corners (`Q` curves with 40px radius at each row-end turn). Translate these to Flutter `Path.quadraticBezierTo()` calls.
- **Pulse animation:** `AnimationController` (repeat, 2.4s) driving a `Container` with a `BoxDecoration` whose `BoxShadow` spread radius lerps from 0 to 14px. No external animation package needed.
- **Mock data shape:** A simple Riverpod provider returning `JourneyProgress(masteredIds: {'alif', 'baa', 'taa'}, currentId: 'thaa')`. The screen computes each letter's state from that.

</specifics>
