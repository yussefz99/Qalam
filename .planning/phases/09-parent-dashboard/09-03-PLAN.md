<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| child (untrusted) → /parent route | The PIN gate + per-entry relock is the only access barrier |
| child → PIN entry field | Shoulder-surf / on-screen leak risk at the input |
| parent screen → progress data | Must be strictly read; no write path exposed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-03 | Elevation of Privilege | /parent access | mitigate | Default-deny gate: screen renders the PIN flow until parentGate.unlocked; relocks on Done/dispose (per-entry, D-07). Verified by parent_gate_test + device check. |
| T-09-04 | Information Disclosure | PIN entry field | mitigate | obscureText:true, numeric keyboard, no autofill/suggestions, never logged (Pitfall 4); generic "Incorrect PIN" — no per-digit oracle (V7). |
| T-09-05 | Spoofing | redirect loop / bypass | mitigate | Synchronous redirect only (no await Drift — Pitfall 2); single /parent route chooses state in-widget; merged refreshListenable re-runs on gate flip — no loop. |
| T-09-09 | Tampering | dashboard write-back | mitigate | Screen wires ONLY read accessors via parentProgressProvider; no edit/delete/reset affordance (read-only hard constraint; widget test asserts absence). |
| T-09-10 | Information Disclosure | error/loading leakage | mitigate | loading/error degrade to the calm empty-state, never a raw stack trace; PIN material never surfaced in any error. |

**Threat-accepted (documented):** Per-entry PIN with no session unlock is intentional (D-07) — more friction than a session token but correct for a shared child device. The 4-digit + PBKDF2 + persisted-cooldown posture is the ratified child-barrier (not device-security) framing from 09-02.
</threat_model>
