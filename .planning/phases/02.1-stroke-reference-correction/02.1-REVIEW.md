# Phase 02.1: Code Review Report

**Reviewed:** 2026-06-01T14:06:21Z
**Depth:** standard
**Files Reviewed:** 17

## Summary

Phase 02.1 corrects the Phase-2 stroke-reference defect (alif authored as a closed
64-point glyph outline) by re-authoring letters as open centerlines and gating the
curriculum load with a pure-Dart validator (`stroke_validation.dart`). The core design
is sound and matches the project's stated invariants:

- **No client-side tutor** â€” none present; correct.
- **Riverpod only** â€” both providers (`curriculumRepository`, `appRouter`) use
  `@Riverpod`/`riverpod_annotation`; no BLoC/GetX. Correct.
- **Child stroke points in-memory only (T-01-05)** â€” the authoring screen holds points
  in `_AuthoredStroke.points` / `_active`, never persists, never logs/`print`s, and
  discards on Clear/dispose. The export path (`authoring_export.dart`) only transforms
  in memory and returns a String. Verified: no `print`/`debugPrint`/file write of point
  data anywhere in scope. (Note: authored data here is curriculum reference data, not
  child trace data â€” the stricter T-01-05 surface is the practice screen, out of scope.)
- **Validator vs. shipped data** â€” confirmed `assets/curriculum/letters.json` has 28
  letters and only `alif` carries strokes: one `line`/`topToBottom`, 5 pts, first
  `[0.5,0.0]` last `[0.5,1.0]`. It passes the validator, consistent with the corrected
  centerline claim.

No BLOCKER-severity defects were found. The findings below are robustness gaps, an
authoring footgun, and test-quality issues that weaken the regression guard's proof.

## Warnings

### WR-01: `validateStroke` can crash (RangeError) on a malformed point instead of returning a violation

**File:** `lib/core/scoring/stroke_validation.dart:94-152`
**Issue:** The RANGE loop defensively tolerates a point that is not an `[x, y]` pair
(`if (p.length != 2) { violations.add(...); continue; }`, lines 96-98). But the
NOT-CLOSED and DIRECTION checks that run afterward access `p[0]`/`p[1]` on *all* points
with no such guard:
- line 140 `_distance(pts.first, pts.last)` and line 141 `_bboxDiagonal(pts)`
- lines 155-157 `final first = pts.first; ... last[0] - first[0]`

If any point has length < 2, `_distance`/`_bboxDiagonal` throw `RangeError` and the
function aborts mid-validation instead of returning a clean violation list. The whole
point of this validator is to surface *human-readable* violations; a crash defeats that
and, via the load-time guard, would surface as an opaque `RangeError` rather than the
intended `StateError` naming the offending letter.

This is not reachable through the JSON load path today (`StrokeSpec.fromJson` always
emits length-2 points and would itself throw earlier on a 1-element pair), so it is a
WARNING, not a BLOCKER. But `validateStroke` is a public, independently-called function
(tests, authoring tool) and the inconsistency between "tolerate at RANGE, crash later"
is a latent defect.
**Fix:** After the RANGE loop, short-circuit if any point is malformed, e.g.:
```dart
// If any point is not a 2-tuple, geometry checks below are unsafe â€” stop here.
if (pts.any((p) => p.length != 2)) return violations;
```
Place this immediately after the RANGE loop (after line 110), before the DOT/NOT-CLOSED/
DIRECTION blocks.

### WR-02: Authoring screen hardcodes `direction: 'topToBottom'` for every multi-point stroke

**File:** `lib/dev/authoring_screen.dart:100-108`
**Issue:** `_endStroke` tags any swipe (â‰Ą2 points) as `type: 'line', direction:
'topToBottom'` regardless of the actual trace direction. For a stroke the owner traces
bottomâ†’top, rightâ†’left, or leftâ†’right (e.g. baa's `rightToLeft` bowl), the exported
fragment will carry a direction that disagrees with the points and **fail the D-04
DIRECTION check** unless the owner remembers to flip the dropdown. The export silently
encodes a wrong default rather than inferring direction from the captured points it
already has. For a tool whose sole job is producing validator-passing fragments, this is
an avoidable footgun.
**Fix:** Infer the default direction from the captured endpoints, mirroring the
validator's own firstâ†’last logic:
```dart
String _defaultDirection(List<Offset> pts) {
  if (pts.length == 1) return 'tap';
  final dx = pts.last.dx - pts.first.dx;
  final dy = pts.last.dy - pts.first.dy;
  if (dy.abs() >= dx.abs()) return dy > 0 ? 'topToBottom' : 'bottomToTop';
  return dx > 0 ? 'leftToRight' : 'rightToLeft';
}
```
Use it in `_endStroke` instead of the constant `'topToBottom'`. The dropdown still lets
the owner override.

### WR-03: The load-time-guard regression test does not isolate the loop check it claims to exercise

**File:** `test/data/curriculum_repository_test.dart:10-32, 204-211`
**Issue:** `_loopLetterJson` uses points `[[0.1,0.1],[0.9,0.1],[0.5,0.9],[0.1,0.1]]` with
`direction: "topToBottom"`. first.y = last.y = 0.1, so `dy = 0` and the DIRECTION check
(`dy > 0`) *already* produces a violation. The test only asserts `throwsA(isA<StateError>())`,
so it would still pass even if the NOT-CLOSED loop detection were broken or deleted â€” the
DIRECTION violation alone makes it throw. The test named for the closed-loop guard does
not actually prove the closed-loop guard fires at load. (The dedicated unit test in
`stroke_spec_validation_test.dart` does cover the loop signature, but the *load-path*
regression here is the one tied to T-02.1-03 and it is not isolating its target.)
**Fix:** Either author the fixture loop with a consistent direction (so DIRECTION passes
and only NOT-CLOSED can fire) using a vertical-going closed shape, or assert on the
thrown message content, e.g. `throwsA(predicate((e) => e.toString().contains('loop')))`,
so a regression in loop detection actually fails this test.

### WR-04: `_InkPainter.shouldRepaint` relies on identity of a list rebuilt every frame; in-place mutation of the active stroke is fragile

**File:** `lib/dev/authoring_screen.dart:85-91, 304`
**Issue:** `_extendStroke` mutates the active point list in place (`active.add(p)`) inside
`setState`, while `_InkPainter.shouldRepaint` compares `old.strokes != strokes` by
reference. This currently repaints only because `_allInk` (lines 80-83) builds a brand-new
outer `List<List<Offset>>` on every `build`, so the reference always differs. The painter
therefore never compares the *contents* of the in-progress stroke â€” repaint correctness is
an accidental side effect of `_allInk` allocating fresh, not of the painter's logic. A
future refactor that memoizes `_allInk` (a natural optimization) would silently stop the
live ink from updating mid-stroke. This is dev-tool-only, hence WARNING.
**Fix:** Make `shouldRepaint` honest about what changed, e.g. compare lengths/contents,
or document explicitly that it depends on `_allInk` returning a fresh list each build.
A minimal robust form:
```dart
@override
bool shouldRepaint(_InkPainter old) => true; // dev tool: capture surface repaints freely
```

## Info

### IN-01: Golden test hardcodes the font family string instead of the `QalamFonts.arabic` token

**File:** `test/curriculum/reference_overlay_golden_test.dart:111`
**Issue:** The overlay golden uses `fontFamily: 'Noto Naskh Arabic'` as a literal, while
the authoring screen uses the `QalamFonts.arabic` constant (authoring_screen.dart:195).
If the token value ever drifts from the literal, the golden backdrop renders as tofu but
still produces a stable (wrong) baseline, masking a real glyph regression.
**Fix:** Import the theme and use `QalamFonts.arabic` so the test tracks the same token
the app renders.

### IN-02: `letter_test.dart` uses semantically invalid stroke fixtures (a "dot" with 2 points / `topToBottom`)

**File:** `test/models/letter_test.dart:27-36`
**Issue:** The `dot` stroke fixture has two points and `direction: 'topToBottom'`. The
model layer doesn't validate (correct, by design â€” validation is the validator's job), so
the test passes, but the fixture encodes data the D-04 validator would reject, which is
misleading to a future reader using this as an example of a dot.
**Fix:** Give the dot one point and `direction: 'tap'`, matching the real schema, so the
fixture doubles as accurate documentation.

### IN-03: `getLetter`/`getLesson` use exception-driven control flow for the not-found case

**File:** `lib/data/curriculum_repository.dart:77-84, 91-98`
**Issue:** Both catch the `StateError` thrown by `firstWhere` to return `null` on a miss.
This works, but throwing-and-catching as normal control flow is a code smell and slightly
obscures intent.
**Fix:** Prefer `firstWhereOrNull` (from `package:collection`) for clarity:
`return letters.firstWhereOrNull((l) => l.id == id);`. Optional polish.

### IN-04: Python tool's nested `resolve_via_glyf` swallows all exceptions and uses a broad `except`

**File:** `tools/extract_reference_paths.py:224-227, 242-243`
**Issue:** `try: ... except Exception: raw_contours = []` (and the inner equivalents)
silently swallow any contour-parsing failure, degrading a single broken glyph to "0
contours" with no diagnostic. For an offline authoring-hint tool this is acceptable, but a
malformed font would silently produce empty hints with no warning to the operator.
**Fix:** Catch the specific expected exception and emit a `WARNING:` line naming the glyph,
so a degraded extraction is visible rather than silent.
