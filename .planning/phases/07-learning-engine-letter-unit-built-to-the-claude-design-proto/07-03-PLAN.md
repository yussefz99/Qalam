<objective>
Build the validator spine: one entry point `validateExercise(exercise, strokes)` that dispatches on
the structured `Check.base` (glyph / sequence / order) and layers the modifiers (positionalForm /
joinContinuity / transformRule), REUSING the existing geometric scorer (`scoreLetter`, Phase 4) for
the glyph case — exactly the "one core scorer + two thin wrappers + a few rule checks" design from
COMPONENT-SYSTEM.md §6 / SCHEMA-V2.md §3. The result is a `CheckResult { passed, mistakeId? }` so
the FeedbackPanel (Plan 07-04) can resolve the matched `mistakeId` to the exercise's authored
`feedback[<mistakeId>]` line, with `feedback['pass']` for the pass case.

Purpose: this is the backend that turns a child's strokes into a pass-or-specific-fix verdict for
EVERY question type, all data-driven from the Exercise config. TDD plan: write the validation
contract tests RED first, then make them GREEN against the real scorer.
Output: the validator + result model + a full RED→GREEN test suite over the baa exercise families.
</objective>
