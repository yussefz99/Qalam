**Topology verdict:** `C-staged`

## Headline

Ship Path A (client-only) as the v2 tutor, re-scoped per the adversarial refutation to "pure-A plus one paragraph of documented foresight." The tutor is a small hand-rolled Dart function-calling loop on firebase_ai 3.13.0 (Gemini Flash, cloud), reaching the Flutter UI through a ~100-line native function-call dispatcher — no AG-UI, no A2UI/genui, no MCP, no server. A single in-process abstraction, TutorBrain (Future<TutorDecision> next(TutorFacts)), unifies THREE LOCAL implementations that all ship in v2: GeminiBrain (cloud), GemmaBrain (flutter_gemma 1.0.2, on-device, experimental, gated on the Phase 13 bake-off), and AuthoredFallback (owner-signed lines, zero-model offline floor). The deterministic scorer owns pass/fail and the mastery star; the agent only chooses words and the next exercise, acting solely through the 4 ACTION tools (present_activity, say, give_hint, advance) with FACTS injected as system-instruction text and a single non-PII payload chokepoint. Path B (a thin agent backend speaking AG-UI/Genkit-flow to Flutter) is NOT built and NOT abstracted-for in code; it is recorded as a documented revisit gate only — because its only Flutter client (ag_ui) is frozen at 0.1.0, 9 months stale, 6 likes, backend-bound, and a remote backend needs a streaming/transport interface that TutorBrain deliberately does not pretend to be.

## Completed adversarial refutation

- **Target:** C-staged (ship Path A now, design the TutorBrain seam so a Path-B backend can slot in later) — **survives:** True
  - Strongest objection: The "cheap A→B migration" that is the SOLE justification for choosing C over pure-A is an unfalsifiable promise that C, as scoped, will not keep — and paying for it now actively harms the Technion demo.

C is identical to pure-A in every shipped artifact. The only thing C adds today is the TutorBrain seam designed "so Path-B can slot in later." But the synthesizer's own load-bearing correction destroys the value of that seam: the FACTS-in/ACTIONS-out interface that is correct for in-process Dart impls (GeminiBrain, GemmaBrain, AuthoredFallback) is the WRONG seam for a remote Path-B backend. The three local impls share a synchronous-ish, single-turn `Future<TutorDecision> next(TutorFacts)` shape with no notion of streaming tokens, SSE event ordering, connection lifecycle, partial/aborted turns, retry-across-network, or App Check token attachment per request. A real RemoteAgUiBrain/RemoteGenkitFlowBrain needs an event-stream contract (AG-UI's 17 event types, or chunked Genkit-flow-over-HTTP), backpressure, cancellation on canvas reset, and a wire schema — none of which `TutorDecision next(facts)` expresses. So when day-B actually comes, you do NOT slot a 4th impl behind the existing interface; you redesign the interface (turn it into a stream, thread cancellation, add transport errors as first-class states) and then touch every call site and every existing impl. That is exactly the rip-out C was sold to avoid. The seam that makes the THREE local impls clean is provably not the seam a remote backend needs — the synthesizer even proves this for the model layer ("cloud and on-device are TWO seams, not one") and then ignores the same logic one layer up at the topology boundary.

Concretely for Qalam on the Technion timeline: choosing C over A buys a migration option whose strike price (re-architecting the interface to a streaming/transport contract) is unchanged by the abstraction you paid for, while the premium is paid now in the one currency you cannot afford — owner comprehension and demo-hardening time. A Dart-new owner who wants low-magic code now reads a 4-impl factory + router + a DTO chokepoint + a "future RemoteBrain" comment, instead of ~30 lines of direct firebase_ai loop. The seam adds indirection that must be explained, tested (the GROUND-02 payload assertion still needed either way), and demoed-around, for a payoff (B) that the migration trigger itself admits requires a parent dashboard / external tools / server memory — none of which exist or are planned in baa-only v2.
  - Conditions: C survives ONLY if it is explicitly RE-SCOPED so that its "design for B" component costs literally zero beyond what pure-A already requires — i.e., C must collapse to "pure-A, plus a one-paragraph ADR documenting the migration trigger and naming Genkit-flow-over-HTTP as the future seam." Survival conditions, all required:

1. STOP CALLING THE TutorBrain INTERFACE THE B-SEAM. The honest design is: TutorBrain is the seam that unifies the three LOCAL impls (Gemini cloud / Gemma on-device / AuthoredFallback) — that is real, load-bearing, and worth building because all three ship in v2. Its justification is the offline floor + the on-device bake-off router, NOT future-B. The B migration is handled by a different (future, streaming/transport) interface, and the ADR must say so plainly rather than implying a 4th impl drops in behind the same `next(facts)`. This removes the false promise without removing any shipped code.

2. THE ONLY B-ARTIFACT NOW IS PROSE. No abstraction, factory hook, transport stub, or "RemoteBrain" placeholder is written for B. The migration trigger (parent dashboard / external tools / server memory) and the recommendation (Genkit-flow-over-HTTP, retire AG-UI-to-Flutter framing) live in the ADR as a documented revisit gate — not in lib/. If any Dart is written today purely to "keep B's door open," C has failed its own cost test and pure-A is strictly better.

3. THE SEAM EARNS ITS KEEP ON LOCAL GROUNDS OR IS DROPPED. Validate that GeminiBrain/GemmaBrain/AuthoredFallback genuinely vary behind one `Future<TutorDecision> next(TutorFacts)` — if the on-device tool-token path (flutter_gemma <tool_call>) or the streamed final-say turn forces the interface to diverge before B even arrives, simplify to direct firebase_ai for the demo and gate Gemma/router to the Phase 13 result. The seam must be justified by the offline floor today, never by B tomorrow.

Net: the RECOMMENDATION (ship A, document B's trigger, two seams not one) is correct and survives. But its STATED RATIONALE ("C wins because A→B migration is cheap if the seam is right") is the weak point and must be downgraded: the migration is cheap because B is a documented future option, NOT because today's local-unifying interface is reusable as B's transport contract. Mislabeling the local seam as the B-seam is the one concrete way C decays into "pay abstraction tax now, still rip out the interface later" — strictly worse than pure-A. Re-scoped to the three conditions above, C is just "pure-A with a paragraph of foresight," which dominates both pure-A (keeps the door honestly noted) and B (no server, no churn, no Dart-new-owner magic) on the Technion timeline.
